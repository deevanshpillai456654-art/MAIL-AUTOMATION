Detailed Findings

1

Inbox layout is broken by conflicting CSS definitions

The inbox markup expects a sidebar, message list, and preview column, but the active theme overrides it to two columns.

Critical

Evidence

  • Fresh screenshot: artifacts/dashboard-visual-smoke/desktop-1440x900/inbox.png
  • The message preview area is visually displaced and the lower-left page area has a large empty block.
  • The smoke test passes because it does not assert actual grid placement.

Source anchors

backend/dashboard/index.html:571 - .inbox-layout contains the inbox layout backend/dashboard/enterprise-ui.css:1580 - defines 3 columns backend/dashboard/hybrid-theme.css:5953 - overrides to 2 columns

Recommended fix: consolidate the inbox grid into one source of truth and make the hybrid theme preserve the required 3-column desktop layout.

2

Sample automation rules still appear as active client rules

Demo rules are visible in the user-facing Rule Lifecycle panel, which conflicts with the client-ready requirement.

Critical

Evidence

  • Fresh screenshot: artifacts/dashboard-visual-smoke/desktop-1440x900/automations.png
  • Visible rules include Quarantine suspected scams, Move invoices to Finance, and Archive old newsletters.
  • These should only appear after a deliberate Load sample rules action.

Source anchors

backend/dashboard/enterprise-ui.js:2299 - loads /api/v1/rules into state.rules backend/dashboard/enterprise-ui.js:2310 - renders every returned rule

Recommended fix: ensure the API excludes is_sample rules by default and add frontend defensive filtering.

3

Rule simulation can create a real rule

The simulate path persists a draft rule before simulating if no existing rule ID is available.

Critical

Impact

  • A user pressing Simulate can unintentionally save a rule.
  • This violates the dry-run requirement: no real emails, rules, labels, folders, or records should be modified.
  • It can also explain why demo or test rules reappear.

Source anchor

backend/dashboard/enterprise-ui.js:2376 - POST /api/v1/rules inside simulateRule()

Recommended fix: add a backend simulate-draft endpoint or simulate from the unsaved draft payload without persisting.

4

Rule lifecycle action buttons are likely no-ops

Pause, Duplicate, and Archive buttons render without action-specific handlers.

High

Impact

  • Users can click visible controls that do nothing.
  • This makes the automations page feel unfinished and unreliable.
  • Client operators cannot confidently pause or archive automation rules.

Source anchor

backend/dashboard/enterprise-ui.js:2310 - buttons have no dataset/action wiring

Recommended fix: wire each action to explicit API endpoints and update state after success.

5

Two dashboard buttons use the same refreshBtn ID.

High

Impact

  • Duplicate IDs are invalid HTML.
  • Selectors, labels, tests, and event delegation become ambiguous.
  • One button may trigger behavior intended for another context.

Source anchors

backend/dashboard/index.html:301 - first refreshBtn backend/dashboard/index.html:326 - second refreshBtn backend/dashboard/enterprise-ui.js:2982 - delegates on target.id === 'refreshBtn'

Recommended fix: use unique IDs or data actions such as data-action="refresh-dashboard".

Critical

Impact

  • Multiple OAuth accounts for the same provider can still be mixed from this panel.
  • This conflicts with provider plus email token isolation.
  • The app can use the first connected provider token for a second mailbox flow.

Source anchors

platform/connectors-panel/frontend/app.js:880 - provider-only startOAuth button platform/connectors-panel/frontend/app.js:908 - startOAuth(provider) platform/connectors-panel/frontend/app.js:909 - /oauth/authorize/${provider}?tenant_id=...

Recommended fix: require email input for OAuth-capable providers and call the mailbox-aware OAuth start endpoint.
